认识着色器

1 为什么着色器很快

着色器之所以很快，主要归功于 GPU 的并行处理能力和硬件加速特性。以下是具体原因：

  • 并行处理能力：GPU 设计有大量简单的微处理器核心，可以同时处理多个任务。在图形渲染中，每个像素都可以被独立计算，这使得 GPU 能够通过并行处理大幅提高效率。例如，在一个 800x600 分辨率的屏幕上，每帧需要处理 480,000 个像素，如果使用传统的单线程 CPU，这些任务将依次排队处理，而 GPU 可以通过其众多的核心同时处理这些像素，从而显著减少处理时间。
  • 硬件加速的特殊数学函数：GPU 内置了专门用于执行特定类型数学运算（如三角函数、矩阵运算）的硬件单元。这些运算在 3D 图形渲染中非常常见，例如坐标变换、光照计算等。由于这些运算可以直接由硬件完成，因此比软件实现要快得多。
  • 针对图形优化的架构：GPU 的架构专门为图形处理进行了优化，它能够高效地管理和调度大量的小型任务，确保数据流顺畅，最大限度地减少延迟和瓶颈

综上所述，着色器利用了 GPU 的强大并行处理能力和硬件加速特性，能够在短时间内完成大量的图形计算

2 什么是 GLSL？

2.1 GLSL（OpenGL Shading Language）

  • 定义：GLSL 是 OpenGL 的着色器编程语言，用于编写顶点着色器和片段着色器等图形处理程序。
  • 监管机构：由 Khronos Group 监管，确保其符合 OpenGL 规范。
  • 特点：
    • 强大的并行处理能力，适用于复杂的图形渲染任务。
    • 支持多种数据类型和内置函数，方便开发者进行高效的图形计算。

2.2 GLSL ES（OpenGL Shading Language Embedded Systems）

  • 定义：GLSL ES 是 GLSL 的简化版本，专为嵌入式系统（如手机、游戏主机）设计。
  • 目标设备：应用于移动设备和其他资源受限的硬件，以降低功耗和减少性能开销。
  • 特点：
    • 删除和简化了部分功能，使其更适合低功耗设备。
    • 保留了核心的图形处理能力，确保在资源有限的情况下仍能高效运行。

2.3 Khronos Group

  • 组织性质：专注于指定开放标准的行业协会，致力于创建免费的 API，使多媒体内容可以在各种平台和设备上得到硬件加速。
  • 主要标准：
    • OpenGL：用于桌面系统的图形 API。
    • OpenGL ES：用于嵌入式系统的图形 API。
    • WebGL：用于网页上的交互式 3D 图形。
  • 官网：Khronos 官网 提供更多关于这些标准和技术的信息。

3 着色器难点

3.1 1. 线程独立性

  • 限制：每个线程必须完全独立于其他线程工作，不能直接与其他线程通信或共享状态。
  • 影响：这意味着你无法检查其他线程的结果、修改输入数据或传递中间结果。这种严格的独立性使得编写复杂的、依赖于全局状态的算法变得非常困难。

3.2 2. 单向数据流

  • 限制：所有数据必须沿同一方向流动，不能回溯或循环。
  • 影响：这限制了某些复杂算法的实现，尤其是那些需要迭代或递归处理的数据结构。开发者必须找到替代方法来实现这些功能，增加了编程的复杂性。

3.3 3. 无记忆性

  • 限制：每个线程在完成任务后不会保留任何状态信息，每次启动时都是全新的。
  • 影响：线程无法记住之前的状态或操作，导致难以实现需要持续状态的应用逻辑。例如，在一个游戏中，你可能希望一个着色器根据之前的帧进行渐变效果，但这在标准着色器中是不可行的，除非通过外部机制（如纹理）传递状态。

3.4 4. 高度抽象化

  • 需求：为了适应并行处理的需求，着色器代码通常需要高度抽象化，编写通用函数来根据位置逐像素改变结果。
  • 挑战：这种抽象化要求开发者具备较高的数学和图形学知识，以及对底层硬件的理解。对于初级程序员来说，这无疑是一个巨大的挑战。

3.5 5. 调试困难

  • 问题：由于并行处理的特性，传统的调试工具和技术往往不适用。错误可能只在特定的线程组合下出现，难以复现和定位。
  • 影响：这使得调试和优化着色器代码变得更加复杂和耗时。

3.6 6. 性能优化

  • 需求：为了充分利用 GPU 的并行处理能力，开发者需要精心设计算法，确保每个线程都能高效运行。
  • 挑战：这不仅需要深厚的算法知识，还需要对硬件架构有深入的理解，以便进行有效的性能优化。

基础语法

1 语言基础

  1. 大小写敏感
    • GLSL ES 是大小写敏感的。例如，Niuniu 和 niuniu 是两个不同的变量。
  2. 语句结束符
    • 每条语句必须以分号 ; 结束。与 JavaScript 不同，GLSL ES 中不能省略分号。
  3. main() 函数
    • 每个着色器程序都必须包含一个 main() 函数，这是程序的入口点。main() 函数不接收任何参数，并且返回类型必须是 void。
  4. 注释
    • 支持单行注释 // 和多行注释 /* ... */。
  5. 数据类型
    • 数值类型：
      • GLSL ES 支持整数类型和浮点数。没有小数点的值被认定为整数类型，而有小数点的则被认为是浮点数。
    • 布尔值类型：
      • 支持 true 和 false 两个布尔值。

2 变量和数据类型

GLSL ES（OpenGL Shading Language for Embedded Systems）支持多种数据类型，这些数据类型可以分为基本类型、向量类型、矩阵类型和结构体类型。

  • 基本数据类型：float, int, bool。
  • 向量类型：如 vec2, vec3, vec4，分别用于浮点、整数和布尔值。
  • 矩阵类型：如 mat2, mat3, mat4，用于变换操作。
  • 结构体类型：用于定义自定义数据类型。
  • 采样器类型：用于访问纹理数据。
  • 数组类型：用于存储多个相同类型的元素。

其中

  1. 变量名规则与其他语言基本一致：只能由 a-z、A-Z、0-9 以及 _ 组成，并且变量名的第一个字符不能为数字
  2. 不能以 gl_、webgl_ 或 _webgl 开头，这些前缀已被 OpenGL ES 保留了。
  3. GLSL ES 是大小写敏感的。例如，Niuniu 和 niuniu 是两个不同的变量。

  • **浮点数 (float)**：

3 类型转换与运行规则

在 GLSL ES 中，基础类型的运算遵循严格的类型匹配规则，没有隐式类型转换。GLSL 使用构造函数进行 显式类型转换。

  1. **float 与 int**：需要显式转换，不能直接运算。
  2. **float 与 vec, mat**：支持逐分量运算，结果保持相同的向量或矩阵类型。
  3. **vec 与 vec**：阶数必须相同，运算为逐分量运算。
  4. **vec 与 mat**：阶数必须相同，乘法遵循线性代数规则。
  5. **mat 与 mat**：阶数必须相同，乘法为矩阵乘法，其他运算为逐分量运算。

逐分量运算是线性的,这就是说 vec 与 float 的运算结果是还是 vec

4 流程控制

GLSL 的流控制语句与 C 语言非常相似，支持 if、for、while、do-while 等常见的控制结构。唯一的显著不同是片段着色器中的 discard 关键字，它用于提前退出片段着色器，不执行后续的片段操作，并且该片段不会写入帧缓冲区。

4.1 discard 关键字

discard 只能在片元着色器中使用，表示放弃当前片元直接处理下一个片元。

5 函数

5.1 函数声明

  • 位置：函数必须在程序的最外部声明，不能嵌套在其他函数内部。
  • 递归调用：GLSL 不允许函数递归调用。
  • 返回值类型：函数必须声明返回值类型，如果没有返回值，则声明为 void。

5.2 内置函数

  • 角度函数：
    • radians：角度转弧度。
    • degrees：弧度转角度。
  • 三角函数：
    • sin：正弦。
    • cos：余弦。
    • tan：正切。
    • asin：反正弦。
    • acos：反余弦。
    • atan：反正切。
  • 指数函数：
    • pow：次方。
    • exp：自然指数。
    • log：对数。
    • sqrt：开平方。
    • inversesqrt：开平方的倒数。
  • 通用函数：
    • abs：绝对值。
    • min：最小值。
    • max：最大值。
    • mod：取余数。
    • sign：取符号。
    • floor：向下取整。
    • ceil：向上取整。
    • clamp：限定范围。
    • fract：获取小数部分。
  • 几何函数：
    • length(x)：计算向量 x 的长度。
    • distance(x,y)：计算向量 xy 之间的距离。
    • dot(x,y)：计算向量 xy 的点积。
    • cross(x,y)：计算向量 xy 的叉积。

6 限定符

6.1 变量限定符

在 GLSL ES 中，变量限定符用于定义变量的作用域和访问权限

就像 JavaScript 和 C 语言，GLSL ES 中也有全局变量和局部变量的概念。全局变量可以在程序中的任意位置使用，而局部变量只能在有限的某一部分代码中使用。

在 GLSL ES 中，如果变量声明在函数的外面，那么它就是全局变量，如果声明在函数内部，那就是局部变量。

  1. 全局变量限制：全局变量的限定符只能是 const、attribute、uniform 和 varying 中的一个，不可复合使用。
  2. 结构体字段：结构体中的字段不能用 const 修饰，但整个结构体变量可以用 const 修饰。
  3. attribute 的使用场景：仅限于顶点着色器，用于从数据缓冲区读取顶点属性数据。
  4. uniform 的使用场景：用于传递外部程序的不变数据，如变换矩阵、光照参数等。
  5. varying 的使用场景：用于在顶点着色器和片段着色器之间传递数据，确保数据一致性。

6.1.1 默认（无限定符）

  • 本地变量，可读可写；函数的输入参数也是这种类型。
  • 局部变量可以在函数内部自由修改。

6.1.2 const

  • 变量初始化后不可变。
  • 局部变量和函数参数可以使用 const 修饰。
  • 结构体变量可以用 const 修饰，但结构体中的字段不能用 const 修饰。

6.1.3 attribute

  • 只能出现在顶点着色器中，只能被声明为全局变量，被用来表示逐顶点的信息。
  • 用于从数据缓冲区读取模型顶点、法线、颜色、纹理等数据

6.1.4 uniform

  • 可以用在顶点着色器和片元着色器中，且必须是全局变量。
  • uniform 变量是只读的，它可以是除了数组或结构体之外的任意类型。
  • 如果顶点着色器和片元着色器中声明了同名的 uniform 变量，那么它就会被两种着色器共享。
  • uniform 变量包含了“一致”(非逐顶点/逐片元的，各顶点或各片元共用) 的数据，JavaScript 应该向其传递此类数据。
  • 用于传递变换矩阵、光照参数等环境数据

6.1.5 varying

  • varying 变量必须是全局变量，它的任务是从顶点着色器向片元着色器传输数据，不能在片元着色器中修改。
  • 必须在两种着色器中声明同名、同类型的 varying 变量
  • 使用它我们可以在顶点着色器中声明一个变量并对其赋值，经过插值处理后，在片元着色器中取出插值后的值来使用。

6.2 函数参数限定符

在 GLSL 中，函数参数可以通过不同的限定符来控制参数的传递方式。这些限定符决定了参数是按值传递还是按引用传递，以及它们在函数内部的行为。

  1. in 和默认行为：in 是默认的参数限定符，意味着如果你不指定任何限定符，参数将以 in 方式传递。
  2. out 的使用：out 参数在函数外部是未初始化的，必须在函数内部赋值，否则会导致未定义行为。
  3. inout 的灵活性：inout 参数允许你在函数内部读取和修改参数，并将修改后的值返回给调用者。
  4. 性能考虑：对于大型数据结构（如数组或复杂结构体），使用 inout 可以避免不必要的拷贝，提高性能。

6.3 精度限定符

在 GLSL 中，为了优化性能和适应不同设备的浮点数运算能力，提供了三种浮点数精度：highp、mediump 和 lowp。这些精度限定符可以用于变量声明，以指定变量的精度

6.3.1 精度限定符

  • highp：最高精度，适用于对精度要求较高的计算。
  • mediump：中等精度，适用于大多数情况。
  • lowp：最低精度，适用于对精度要求不高的计算。

6.3.3  精度确定规则

变量的精度确定规则如下：

  1. 精度限定符：如果变量有精度限定符，则使用该限定符指定的精度。
  2. 右侧表达式：如果没有精度限定符，则查找右侧表达式中已确定精度的变量，整个表达式将使用该精度。
  3. 多个精度：如果找到多个精度限定符，则选择精度较高的那种。
  4. 默认精度：如果找不到任何精度限定符，则使用默认精度或更大的精度类型。

6.3.4 invariant 关键字

由于着色器在编译时会进行优化，可能会导致相同运算在不同着色器中的结果不一致。为了解决这个问题，可以使用 invariant 关键字来确保计算结果的精确一致性。

6.3.5 限定符顺序

当需要使用多个限定符时，应遵循以下顺序：

  1. 一般变量：invariant > storage > precision
  2. 参数：storage > parameter > precision

7.1 常用预编译指令

  • **#define**：定义宏。
  • **#undef**：取消定义宏。
  • **#if**：条件编译，如果条件为真，则编译后续代码。
  • **#ifdef**：如果宏已定义，则编译后续代码。
  • **#ifndef**：如果宏未定义，则编译后续代码。
  • **#else**：与 #if 或 #ifdef 配合使用，表示否则的情况。
  • **#elif**：与 #if 或 #ifdef 配合使用，表示否则如果的情况。
  • **#endif**：结束条件编译块。
  • **#error**：生成编译错误信息。
  • **#pragma**：设置编译器选项或指示编译器执行特定操作。
  • **#extension**：启用或禁用特定扩展。
  • **#version**：指定 GLSL 版本。
  • **#line**：修改当前行号和文件名。

7.2 内置的宏

  • **__LINE__**：当前源码中的行号。

8.2 内置的常量

GLSL 提供了一些内置常量，用于描述当前系统的一些特性。了解这些常量有助于优化着色器程序，提高兼容性和性能。

8.2.1 顶点着色器（Vertex Shader）中的内置常量

  1. gl_MaxVertexAttribs
    • 类型：const mediump int
    • 含义：表示在顶点着色器中可用的最大 attribute 数。
    • 最低要求：至少为 8。
    • 用途：确保顶点属性的数量不超过硬件限制。
  2. gl_MaxVertexUniformVectors
    • 类型：const mediump int
    • 含义：表示在顶点着色器中可用的最大 uniform 向量数。
    • 最低要求：至少为 128。
    • 用途：确保传递给顶点着色器的 uniform 变量数量不超过硬件限制。
  3. gl_MaxVaryingVectors
    • 类型：const mediump int
    • 含义：表示在顶点着色器中可用的最大 varying 向量数。
    • 最低要求：至少为 8。
    • 用途：确保传递给片元着色器的 varying 变量数量不超过硬件限制。
  4. gl_MaxVertexTextureImageUnits
    • 类型：const mediump int
    • 含义：表示在顶点着色器中可用的最大纹理单元数（贴图）。
    • 最低要求：至少为 0（某些设备可能不支持顶点纹理）。
    • 用途：确保顶点着色器中使用的纹理单元数量不超过硬件限制。
  5. gl_MaxCombinedTextureImageUnits
    • 类型：const mediump int
    • 含义：表示在顶点着色器和片元着色器中总共最多支持的纹理单元数。
    • 最低要求：至少为 8。
    • 用途：确保整个渲染管线中使用的纹理单元总数不超过硬件限制。

8.2.2 片元着色器（Fragment Shader）中的内置常量

  1. gl_MaxTextureImageUnits
    • 类型：const mediump int
    • 含义：表示在片元着色器中能访问的最大纹理单元数。
    • 最低要求：至少为 8。
    • 用途：确保片元着色器中使用的纹理单元数量不超过硬件限制。
  2. gl_MaxFragmentUniformVectors
    • 类型：const mediump int
    • 含义：表示在片元着色器中可用的最大 uniform 向量数。
    • 最低要求：至少为 16。
    • 用途：确保传递给片元着色器的 uniform 变量数量不超过硬件限制。
  3. gl_MaxDrawBuffers
    • 类型：const mediump int
    • 含义：表示可用的 drawBuffers 数。
    • 最低要求：在 OpenGL ES 2.0 中为 1，在将来的版本可能会有所变化。
    • 用途：确保多渲染目标（MRT）的数量不超过硬件限制。
