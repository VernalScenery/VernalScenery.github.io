这里主要注意一下require的查找规则，依据 X的不同，查找规则也不同

2.5.1 情况一：X 是一个 Node 核心模块

  • 行为：直接返回核心模块，并且停止查找。

2.5.2 情况二：X 以 ./ 或 ../ 或 /（根目录）开头

  1. 第一步：将 X 当做一个文件在对应的目录下查找。
    • 如果有后缀名，按照后缀名的格式查找对应的文件。
      • 例如：require('./abc.js') 会查找 abc.js 文件。
    • 如果没有后缀名，会按照如下顺序查找：
      1. 直接查找文件 X。
      2. 查找 X.js 文件。
      3. 查找 X.json 文件。
      4. 查找 X.node 文件。
      • 例如：require('./abc') 会依次查找 abc, abc.js, abc.json, abc.node。
  2. 第二步：如果没有找到对应的文件，将 X 作为一个目录。
    • 查找目录下面的 index 文件。
      1. 查找 X/index.js 文件。
      2. 查找 X/index.json 文件。
      3. 查找 X/index.node 文件。
      • 例如：require('./abc') 如果没有找到 abc 文件，则会查找 abc/index.js, abc/index.json, abc/index.node。
  3. 如果没有找到，则报错：not found。

2.5.3 情况三：直接是一个 X（没有路径），并且 X 不是一个核心模块

  • 行为：按照 module.paths 列表中的路径进行查找。

  • 具体步骤：

    1. 从当前目录的 node_modules 文件夹开始查找。
    2. 如果没有找到，继续查找父目录的 node_modules 文件夹。
    3. 依次向上级目录查找，直到根目录。
  • 如果在 d:\100_Code\110_Codeing\webBlog\sourceCode\JS_learn\30_JS模块化解析\02_CommonJS\04_require细节\node_modules\axios 找不到，则继续向上查找父目录的 node_modules 文件夹。
  • 如果没有找到，则报错：not found。

2.5.4 总结

  • 核心模块：直接返回。
  • 相对路径或绝对路径：
    • 先查找文件，按照后缀名顺序。
    • 再查找目录下的 index 文件，按照后缀名顺序。
    • 找不到则报错。
  • 非路径且非核心模块：
    • 按照 module.paths 列表中的路径查找。
    • 找不到则报错。

2.6 模块的加载过程

2.6.1 结论一：模块在被第一次引入时，模块中的 JavaScript 代码会被运行一次

  • 解释：当使用 require 引入一个模块时，如果该模块之前未被加载过，Node.js 会执行该模块中的所有代码，并将其导出的对象缓存起来。

2.6.2 结论二：模块被多次引入时，会缓存，最终只加载（运行）一次

  • 解释：每个模块对象 module 都有一个属性 loaded，用于标识该模块是否已经被加载。
    • loaded 为 false 表示模块还没有加载。
    • loaded 为 true 表示模块已经加载。
  • 当模块被多次引入时，Node.js 会检查缓存，如果模块已经存在于缓存中，则直接返回缓存的对象，而不会再次运行模块中的代码。
为什么只会加载运行一次 ？
  • 缓存机制：Node.js 使用一个缓存系统来存储已经加载过的模块。这个缓存系统是一个对象，键是模块的绝对路径，值是模块对象。
  • 加载过程：
    1. 当 require 被调用时，Node.js 首先检查缓存。
    2. 如果模块已经在缓存中，则直接返回缓存的对象。
    3. 如果模块不在缓存中，则加载并执行模块代码，然后将模块对象存入缓存。

2.6.3 结论三：如果有循环引入，那么加载顺序是什么？

  • 解释：循环引入是指两个或多个模块相互引入对方的情况。这种情况下，Node.js 会按照特定的顺序加载模块，以避免死锁或重复加载。
加载顺序
  • 图结构：模块之间的依赖关系可以表示为一个图结构。
  • 遍历算法：Node.js 采用深度优先搜索（DFS, Depth First Search）算法来遍历和加载模块。
  • Node采用的是深度优先算法：main -> aaa -> ccc -> ddd -> eee ->bbb

先把 aaa这个节点里面的找完，再去查找 bbb里面的

2.7 CommonJS规范缺点

2.7.1 CommonJS 加载模块是同步的

  • 同步加载：CommonJS 规范要求模块加载是同步的，这意味着只有等到对应的模块加载完毕，当前模块中的内容才能被运行。
  • 服务器环境：在服务器环境中，加载的 JavaScript 文件通常是本地文件，加载速度非常快，因此同步加载不会带来明显的性能问题。
  • 浏览器环境：在浏览器环境中，加载 JavaScript 文件需要从服务器下载，这可能会导致延迟。如果采用同步加载，后续的 JavaScript 代码将无法正常运行，即使是一些简单的 DOM 操作也会受到影响。

2.7.2 浏览器中的模块化解决方案

  1. **AMD (Asynchronous Module Definition)**：
    • 异步加载：AMD 规范允许模块异步加载，适用于浏览器环境。
    • 典型实现：RequireJS 是一个典型的 AMD 实现库。
    • 优点：可以并行加载多个模块，提高加载效率。
  2. **CMD (Common Module Definition)**：
    • 按需加载：CMD 规范也是异步加载，但更强调按需加载。
    • 典型实现：SeaJS 是一个典型的 CMD 实现库。
    • 优点：按需加载，减少初始加载时间。
  3. **ES Modules (ESM)**：
    • 现代浏览器支持：现代浏览器已经原生支持 ES Modules，可以使用 import 和 export 语法进行模块化开发。
    • 动态导入：ES Modules 支持动态导入（import()），可以实现按需加载。
    • 优点：标准统一，无需额外工具转换。
  4. Webpack 等构建工具：
    • 代码转换：Webpack 等构建工具可以将 CommonJS 或者 ES Modules 代码转换为浏览器可以直接执行的代码。
    • 优化：通过代码分割、懒加载等技术，进一步优化加载性能。

3 ES Module

3.1 ES Module 和 CommonJS 的不同之处

3.1.1 关键字差异

  • CommonJS：
    • 使用 require 导入模块。
    • 使用 module.exports 或 exports 导出模块。
  • ES Module：
    • 使用 import 导入模块。
    • 使用 export 导出模块。

3.1.2 加载方式

  • CommonJS：
    • 同步加载：CommonJS 模块在服务器端使用，模块加载是同步的。
    • 运行时加载：模块在运行时加载，可以动态加载模块
    • 动态加载：虽然 CommonJS 主要是同步加载，但它也支持动态加载模块，通常通过 require 函数在运行时按需加载模块。
  • ES Module：
    • 异步加载：ES Module 模块在浏览器端使用，模块加载是异步的。
    • 编译时加载：模块在编译时加载，可以进行静态分析。
    • 动态加载：ES Module 也支持动态加载模块，通常通过 import() 函数在运行时按需加载模块。
静态分析

静态分析是指在代码执行之前，通过对代码的语法和结构进行分析，而不实际运行代码，来确定代码的行为和特性。在 ES Module 中，静态分析主要用于确定模块的依赖关系和导入导出的符号。

静态分析的作用

  1. 确定依赖关系：在编译时，JavaScript 引擎可以通过静态分析确定一个模块依赖于哪些其他模块。这有助于构建模块的依赖图，确保在运行时所有依赖的模块都已加载。
  2. 解析导入导出：静态分析可以解析 import 和 export 语句，确定模块之间共享的符号（变量、函数等）。
  3. 优化加载：通过静态分析，可以提前确定哪些模块需要加载，从而优化加载过程，提高应用的启动速度。

静态分析的特点

  1. 编译时确定：静态分析在编译时进行，而不是在运行时。这意味着所有 import 语句必须位于模块的顶层，不能放在条件语句或其他逻辑代码中。
  2. 语法解析：静态分析主要依赖于语法解析，而不是实际执行代码。因此，它不会考虑运行时的条件分支或动态行为。
  3. 确定性：静态分析的结果是确定性的，即在编译时可以确定模块的依赖关系和导入导出的符号。

3.1.3 缓存机制

  • CommonJS：
    • 模块加载后会缓存，多次 require 只会加载一次。
  • ES Module：
    • 模块加载后也会缓存，多次 import 只会加载一次。

3.1.4 严格模式

  • ES Module：
    • 自动采用严格模式（use strict），不需要显式声明。
  • CommonJS：
    • 需要显式声明 use strict

3.5 export和import结合使用

在开发和封装功能库时，通常希望将所有暴露的接口集中管理在一个文件中，以便于统一接口规范和方便阅读。这种情况下，可以使用 export 和 import 结合的方式。

  1. 统一接口规范：将所有导出的接口集中在同一个文件中，可以确保接口的一致性和规范性。
  2. 方便阅读和维护：开发者只需要查看一个文件即可了解整个库提供的所有功能，便于维护和扩展。
  3. 模块化管理：每个功能模块可以独立开发和测试，最终通过一个入口文件统一导出，提高代码的可复用性和可维护性。

以下是几种常见的结合使用方式及其应用场景：

3.5.1 方式一：先导入再导出

首先从其他模块中导入需要的标识符，然后再导出这些标识符

3.5.2 方式二：直接导出

在导出时直接指定从哪个模块导入的标识符。

3.5.3 方式三：使用 export * 导出所有内容

使用 export * from '模块' 语法将一个模块中的所有导出内容重新导出。

这种方式最简洁，适合将一个模块中的所有内容重新导出，在一些库中用的也比较多

3.6 default用法

在 ES6 模块化开发中，除了命名导出（named exports）之外，还支持默认导出（default export）。默认导出的特点是：

  • 导出时不需要指定名字：可以导出任何类型的值（变量、函数、类等）。
  • **导入时不需要使用 {}**：可以直接导入，并且可以自定义导入的名称。
  • 一个模块中只能有一个默认导出：确保了默认导出的唯一性。

导入默认导出时，不需要使用 {}，并且可以自定义导入的名称。

注意：

  • 默认导出的唯一性：一个模块中只能有一个默认导出，否则会报错。
  • 命名冲突：在导入多个模块时，如果多个模块都有默认导出，可以通过自定义导入名称来避免命名冲突。

3.7 静态导入与动态导入

3.7.1 静态导入

在 ES6 模块化系统中，静态导入（如 import 语句）要求在编译时就能确定模块的依赖关系。这意味着所有的 import 语句必须位于模块的顶层，不能放在条件语句或其他逻辑代码中。这是因为 JavaScript 引擎在解析模块时，需要提前知道所有的依赖关系，以确保模块的正确加载和执行。

静态导入的限制

  1. 必须在顶层：import 语句必须位于模块的顶层，不能放在函数、条件语句或其他逻辑代码中。
  2. 编译时确定：依赖关系必须在编译时确定，不能在运行时动态改变。

3.7.2 动态导入（Dynamic Import）

为了克服静态导入的限制，ES11 引入了 import() 函数，允许在运行时动态加载模块。import() 函数返回一个 Promise，可以在异步代码中使用。

动态导入的优势

  1. 运行时确定：可以在运行时根据条件动态选择加载的模块。
  2. 按需加载：可以根据需要懒加载模块，提高应用的性能。
  3. 灵活性：可以在任何地方使用 import()，包括函数、条件语句等。

3.8 ES Module的解析流程

ES Module 在浏览器中的解析过程可以分为三个主要阶段：构建（Construction）、实例化（Instantiation）和运行（Evaluation）。下面详细解释每个阶段的具体内容：

3.8.1 构建（Construction）

  • 目标：根据模块的地址查找并下载 JavaScript 文件，将其解析成模块记录（Module Record）。
  • 步骤：
    1. 解析 <script> 标签：浏览器首先解析 HTML 文件中的 <script> 标签，识别出带有 type="module" 属性的脚本。
    2. 发起请求：根据 <script> 标签中的 src 属性，浏览器发起 HTTP 请求，下载指定的 JavaScript 文件。
    3. 解析文件：下载完成后，浏览器解析 JavaScript 文件，生成模块记录（Module Record）。模块记录包含了模块的源代码、导入和导出信息等。

3.8.2 实例化（Instantiation）

  • 目标：对模块记录进行实例化，分配内存空间，解析模块的导入和导出语句，将模块指向对应的内存地址。
  • 步骤：
    1. 解析导入和导出：浏览器解析模块记录中的 import 和 export 语句，确定模块之间的依赖关系。
    2. 构建依赖图：根据导入和导出信息，构建模块之间的依赖图。这一步确保所有依赖的模块都被正确解析和加载。
    3. 分配内存：为每个模块分配内存空间，创建模块的执行上下文。
    4. 绑定导入和导出：将模块的导入和导出绑定到对应的内存地址，确保模块之间的引用关系正确无误。

3.8.3 运行（Evaluation）

  • 目标：运行模块代码，计算值，并将值填充到内存地址中。
  • 步骤：
    1. 执行模块代码：按照依赖图的顺序，依次执行每个模块的代码。确保在执行某个模块之前，其所有依赖的模块已经执行完毕。
    2. 计算值：执行模块代码时，计算导出值，并将其填充到内存地址中。
    3. 完成执行：所有模块代码执行完毕后，模块之间的引用关系已经建立，模块可以正常工作。

导出模块可以更新变量值

  • 当一个模块导出一个变量时，这个变量实际上是一个绑定（binding）。这意味着导出模块可以在任何时候更新这个变量的值。

例如，在 foo.js 中，name 和 age 被导出后，可以在 foo.js 内部通过 setTimeout 更新它们的值。

导入模块不能更新变量值

  • 导入模块只能读取这些变量的当前值，而不能修改它们。这是因为导入的变量是只读的。

例如，在 bar.js 中导入 foo.js 中的 name 和 age 后，尝试修改这些变量会导致错误

原因：导入的变量是只读的，是为了保证模块之间的隔离性和数据的完整性。如果允许导入模块修改导出变量的值，可能会导致模块之间的依赖关系变得复杂和不可预测。

4 CommonJS 和 ES Module 相互引用

  • 在现代 JavaScript 开发中，CommonJS 和 ES Module 是两种不同的模块系统。CommonJS 主要用于 Node.js 环境，而 ES Module 是 JavaScript 标准的一部分，适用于浏览器环境。
