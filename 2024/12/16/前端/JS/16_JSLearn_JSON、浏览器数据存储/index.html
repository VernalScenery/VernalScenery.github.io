3 JSON 序列化

3.1 基本使用

在ES5中引用了JSON全局对象，该对象有两个常用的方法：

  • stringify方法：将JavaScript类型转成对应的JSON字符串；
  • parse方法：解析JSON字符串，转回对应的JavaScript类型；

3.2 Stringify细节

JSON.stringify 方法用于将 JavaScript 对象或值转换为 JSON 字符串。它有三个参数：

  1. **value**：要转换的 JavaScript 值（通常是对象或数组）。
  2. **replacer**（可选）：一个函数或数组，用于转换结果。
  3. **space**（可选）：用于控制缩进和空格，使生成的 JSON 字符串更易读。

3.2.1  replacer 参数

replacer 参数可以是一个函数或一个数组，用于控制哪些属性被包含在最终的 JSON 字符串中，以及如何转换这些属性的值。

传入数组

当 replacer 是一个数组时，只有数组中列出的属性会被包含在最终的 JSON 字符串中。

传入回调函数

当 replacer 是一个函数时，该函数会在每个属性被序列化时被调用。函数接收两个参数：key 和 value，并返回一个新的值来替换原来的值

3.2.2  space 参数

space 参数用于控制生成的 JSON 字符串的缩进和空格，使其更易读。它可以是一个数字或一个字符串。

数字

当 space 是一个数字时，它表示每个缩进级别使用的空格数。最大值为 10。

字符串

当 space 是一个字符串时，该字符串会被用作缩进字符。

3.2.3 对象中的 toJSON 方法

在 JavaScript 中，如果一个对象包含 toJSON 方法，那么在使用 JSON.stringify 方法时，会优先调用该对象的 toJSON 方法，并使用其返回值作为序列化的结果。

3.3 parse细节

JSON.parse 方法用于解析 JSON 字符串，并将其转换为 JavaScript 值或对象。该方法接受两个参数：

  1. **text**：要解析的 JSON 字符串。
  2. **reviver**（可选）：一个函数，用于在返回之前对所得到的对象执行变换（操作）。

3.3.1 使用 reviver 函数

reviver 函数允许你在解析过程中对每个键值对进行处理。reviver 函数接收两个参数：key 和 value，并返回一个新值来替换原来的值。

假设我们希望将所有字符串类型的值转换为大写：

reviver 函数会在解析过程中递归地调用，处理每一个键值对。具体来说：

  • 对于根对象，key 为空字符串 ""，value 为整个解析后的对象。
  • 对于嵌套对象和数组，key 为当前键，value 为对应的值。

4 使用JSON序列化深拷贝

在 JavaScript 中，使用 JSON.stringify 和 JSON.parse 可以实现对象的深拷贝。但是需要注意，因为 JSON 语法的限制，这种方法有一些限制，特别是对于函数和其他特殊类型的值。

4.1 引用赋值

引用赋值只是创建了一个新的引用，指向同一个对象。修改其中一个对象会影响另一个对象

4.2 浅拷贝

浅拷贝只会复制对象的第一层属性，嵌套对象仍然共享引用。

4.3 深拷贝（使用 JSON.stringify 和 JSON.parse）

JSON.stringify 和 JSON.parse 组合使用可以实现深拷贝，但有一些限制：

  • 函数：JSON.stringify 不会序列化函数，因此深拷贝后的新对象中不会包含函数。
  • 循环引用：JSON.stringify 无法处理对象的循环引用，会导致错误。
  • 特殊值：undefined 和 Symbol 类型的值会被忽略。

浏览器数据存储

1 Web Storage

Web Storage 是一种在浏览器中存储数据的机制，提供了比 cookie 更简单和直观的 key-value 存储方式。

1.1 两种类型

Web Storage 主要分为两种类型：

  1. **localStorage**：本地存储，提供的是一种永久性的存储方法，在关闭掉网页重新打开时，存储的内容依然保留；
    • 提供的是一种永久性的存储方法。
    • 在关闭掉网页重新打开时，存储的内容依然保留。
    • 没有过期时间，除非手动清除或覆盖。
  2. **sessionStorage**：会话存储，提供的是本次会话的存储，在关闭掉会话时，存储的内容会被清除；
    • 提供的是本次会话的存储。
    • 在关闭掉会话（通常是关闭浏览器标签页或窗口）时，存储的内容会被清除
    • 会话存储的数据只在当前会话中有效。

2 IndexedDB

2.1 概览

  • IndexedDB简介：
    • IndexedDB 是一种在客户端存储大量结构化数据的底层 API。
    • 它允许 Web 应用程序存储和检索大量数据，并且可以进行索引以提高查询性能。
  • 特点：
    • 事务型数据库：IndexedDB 是一种事务型数据库系统，所有对数据库的操作都必须在事务中进行。
    • 面向对象：IndexedDB 是一种面向对象的数据库，可以存储复杂的数据结构。
    • 非关系型数据库：类似于 NoSQL 数据库，IndexedDB 不支持 SQL 查询，而是通过 JavaScript API 进行数据操作。
  • 应用场景：
    • 大量数据存储：适用于需要在客户端存储大量数据的场景，如离线应用、缓存数据等。
    • 结构化数据：适合存储结构化数据，如用户信息、文章、评论等。
  • 基本操作：
    • 打开数据库：使用 indexedDB.open 方法打开或创建数据库。
    • 创建对象存储：在 onupgradeneeded 事件中创建对象存储（类似于表）。
    • 事务管理：通过 db.transaction 方法创建事务，指定读写模式。
    • 数据操作：在事务中使用对象存储的 add、get、put 和 delete 方法进行数据的增删改查。

3 Cookie

3.1 认识 Cookie

  • Cookie 简介：
    • 定义：Cookie 是一种小型文本文件，用于在客户端存储数据，以便网站能够识别用户身份。
    • 用途：常用于会话管理、个性化设置、跟踪用户行为等。
  • Cookie 的存储位置：
    • 内存 Cookie：
      • 存储在浏览器的内存中。
      • 浏览器关闭时，内存 Cookie 会被自动删除。
      • 适用于临时会话数据。
    • 硬盘 Cookie：
      • 存储在用户的硬盘中。
      • 有一个过期时间，只有在过期时间到达或用户手动清理时才会被删除。
      • 适用于持久化的会话数据或用户偏好设置。
  • 如何判断 Cookie 类型：
    • 内存 Cookie：
      • 没有设置过期时间（Expires 或 Max-Age 属性未设置）。
      • 关闭浏览器时会自动删除。
    • 硬盘 Cookie：
      • 设置了过期时间（Expires 或 Max-Age 属性已设置），并且过期时间不为 0 或负数。
      • 需要手动清理或等待过期时间到达后才会被删除。

当服务器在响应头中包含 Set-Cookie 字段时，浏览器会自动将该 Cookie 保存在本地，并根据 Expires 属性设置过期时间

当浏览器后续向同一域名发送请求时，会自动在请求头中携带之前保存的 Cookie。

可以通过 document.cookie 查看当前页面的 Cookie。

3.2.2 Cookie的作用域

  • Cookie 的作用域：
    • 作用域：Cookie 的作用域决定了哪些 URL 可以接收和发送该 Cookie。
    • 主要属性：
      • Domain：指定哪些主机可以接受该 Cookie。
      • Path：指定主机下的哪些路径可以接受该 Cookie。
  • Domain 属性：
    • 默认值：如果不指定 Domain，默认是设置 Cookie 的原始域名（origin），不包括子域名。
    • 指定子域名：如果指定了 Domain，则包括子域名。例如，设置 Domain=mozilla.org，则 developer.mozilla.org 也会接收到该 Cookie。
  • Path 属性：
    • 默认值：如果不指定 Path，默认是设置 Cookie 的路径。
    • 指定路径：如果指定了 Path，则只有该路径及其子路径下的请求会携带该 Cookie。例如，设置 Path=/docs，则 /docs、/docs/Web/ 和 /docs/Web/HTTP 都会匹配。

注意事项：

  • Domain 和 Path 必须匹配：如果在设置 Cookie 时指定了 Domain 和 Path，则在删除 Cookie 时也需要指定相同的 Domain 和 Path，否则无法正确删除。

3.3 Cookie 的缺点

  • 存储容量有限：
    • 每个域名下的 Cookie 总大小通常限制在 4KB 左右。这限制了可以存储的数据量。
  • 性能影响：
    • 每次 HTTP 请求都会携带 Cookie，这会增加请求的大小，尤其是在有多个 Cookie 的情况下，可能会影响页面加载速度。
  • 缺乏加密：
    • 默认情况下，Cookie 是以明文形式存储和传输的。虽然可以使用 Secure 标志来确保 Cookie 仅通过 HTTPS 传输，但仍然需要额外的加密措施来保护敏感数据。
  • 存储位置限制：
    • Cookie 存储在客户端，容易被用户手动删除或禁用，这可能导致依赖 Cookie 的功能失效。
  • 兼容性问题：
    • 不同浏览器对 Cookie 的支持和行为可能存在差异，这可能导致跨浏览器兼容性问题。
