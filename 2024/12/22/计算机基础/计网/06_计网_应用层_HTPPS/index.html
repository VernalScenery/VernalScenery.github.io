1 认识 HTTPS

HTTPS（HyperText Transfer Protocol Secure）是一个通过计算机网络进行安全通信的传输协议。它由HTTP和TLS/SSL协议构建，以提供加密的、可靠的通信。

  • 加密：HTTPS使用SSL/TLS加密来保护数据传输的安全性，防止中间人攻击。
  • 认证：确保网站的身份是真实的，防止用户访问到伪造的网站。
  • 完整性：确保数据在传输过程中没有被篡改。

SSL（Secure Sockets Layer，安全套接层）是一种用于在客户端和服务器之间建立加密链接的安全协议。SSL 的主要功能是确保数据传输的安全性，防止数据被窃听或篡改。

SSL 已经被更安全的 TLS（Transport Layer Security，传输层安全）协议所取代。TLS 是 SSL 的继任者，提供了更好的安全性。尽管人们仍然习惯性地使用“SSL”这个术语，但实际上大多数现代实现都使用的是 TLS。

  • 加密通信：SSL 使用对称加密和非对称加密相结合的方式，确保数据在传输过程中不会被第三方窃取。
  • 身份验证：通过数字证书验证服务器的身份，确保客户端连接到的是合法的服务器。
  • 数据完整性：使用消息认证码（MAC）确保数据在传输过程中没有被篡改。

SSL/TLS 握手过程

  1. 客户端问候（Client Hello）：客户端向服务器发送支持的加密算法列表和其他信息。
  2. 服务器问候（Server Hello）：服务器选择一种加密算法，并返回其证书和公钥。
  3. 密钥交换：客户端生成一个预主密钥（pre-master secret），并用服务器的公钥加密后发送给服务器。
  4. 计算会话密钥：双方根据预主密钥计算出会话密钥。
  5. 加密通信：双方使用会话密钥进行加密通信。

2 对称加密和非对称加密

2.1 对称加密

2.1.1 秘钥的传输问题

对称密钥的传输问题是一个重要的安全挑战，因为如果对称密钥在传输过程中被截获，攻击者就可以解密所有使用该密钥加密的数据。为了解决这个问题，通常会结合非对称加密（公钥加密）和对称加密来确保密钥的安全传输。

2.2 非对称加密

  • 公钥和私钥是一对，是一一对应的
  • 使用公钥加密，必须使用对应的私钥才能解密
  • 使用私钥加密的密文，也必须使用对应的公钥才能解密

  • 非对称加密算法有：RSA、DH等
  • 非对称加密的加解密速度比对称加密要慢

2.2.1 秘钥的传输问题

客户端和服务器都有可能作为接收端，故都需要生成自己的公钥和私钥

2.3 混合加密

混合加密（Hybrid Encryption）结合了对称加密和非对称加密的优点，以实现高效且安全的数据传输。它通过使用非对称加密来安全地传输对称密钥，然后使用对称加密来加密大量数据。

2.3.1 混合加密的工作原理

  1. 生成对称密钥：
    • 客户端生成一个随机的对称密钥（如 AES 密钥），用于加密实际数据。
  2. 加密对称密钥：
    • 客户端使用服务器的公钥（非对称加密）加密生成的对称密钥。这一步确保只有拥有对应私钥的服务器才能解密并获得对称密钥。
  3. 加密数据：
    • 客户端使用对称密钥加密需要传输的数据。由于对称加密算法（如 AES）效率高，因此适合加密大量数据。
  4. 传输加密后的数据和加密后的对称密钥：
    • 客户端将加密后的数据和加密后的对称密钥一起发送给服务器。
  5. 解密对称密钥：
    • 服务器使用自己的私钥解密加密后的对称密钥，恢复出原始的对称密钥。
  6. 解密数据：
    • 服务器使用恢复出的对称密钥解密加密后的数据，恢复出原始数据。

3 数字签名

数字签名是一种用于验证数字信息真实性和完整性的技术。它确保信息在传输过程中没有被篡改，并且可以确认信息的发送者身份。

3.1.1 数字签名的工作原理

  1. 生成哈希值：
    • 发送者使用哈希函数（如 SHA-256）对消息生成一个固定长度的哈希值（摘要）。
  2. 加密哈希值：
    • 发送者使用自己的私钥对哈希值进行加密，生成数字签名。
  3. 附加签名：
    • 发送者将数字签名附加到原始消息中，然后发送给接收者。
  4. 验证签名：
    • 接收者使用发送者的公钥解密数字签名，恢复出哈希值。
    • 接收者对收到的消息生成哈希值，并与解密后的哈希值进行比较。
    • 如果两个哈希值匹配，则确认消息的真实性和完整性。

相较于混合加密，客户端这里不仅生成了对称秘钥， 还生成一个非对称秘钥

  • 私钥用于加密散列值，公钥传给服务器解密散列值
  • 因为私钥和公钥是一对，这样也可以确保消息是同一个客户端发送过来的

这里计算的是未加密的消息的散列值

散列值比对一致则表明消息是对方发过来的，也没有被篡改过

  • 同一客户端发的：接收端使用发送端公钥能解密签名值，表明收到的就是发送端的消息
  • 传输过程没被篡改过：散列值一致则保证了消息没有被篡改过

4 证书

4.1 解决的问题

服务器身份验证：证书用于验证服务器的身份，确保客户端连接的是正确的服务器，而不是冒名顶替的钓鱼网站

4.2 证书注册和签发

一个典型的 X.509 证书包含以下信息：

  • 版本：证书的版本号。
  • 序列号：证书的唯一标识符。
  • 签名算法：用于签名证书的算法。
  • 发行者：签发证书的证书颁发机构（CA）。
  • 有效期：证书的有效期，包括开始日期和结束日期。
  • 主体：证书所代表的实体（如服务器、用户）。
  • 公钥：实体的公钥。
  • 扩展：可选的扩展字段，提供额外的信息（如用途、密钥用法等）。
  • 签名：CA 使用其私钥对证书内容进行签名，确保证书的完整性和真实性。

服务器生成公钥和私钥后，需要通过以下步骤进行证书注册和签发：

  1. 生成公钥和私钥：使用非对称加密算法生成公钥和私钥。
  2. 创建证书签名请求（CSR）：包含公钥和一些基本信息。
  3. 提交 CSR 到证书颁发机构（CA）：CA 进行验证。
  4. CA 验证信息并签发证书：CA 使用其私钥对证书进行数字签名。
  5. 安装证书：将签发的证书安装在服务器上，配置 Web 服务器使用该证书。

4.3 对证书进行数字签名的过程

  1. 生成证书的散列值：
    • 首先，CA 对证书的内容（不包括签名部分）计算一个散列值。
    • 常用的散列算法包括 SHA-256、SHA-384 等。
  2. 使用 CA 的私钥进行签名：
    • CA 使用其私钥对生成的散列值进行加密。这个加密后的散列值就是数字签名。
    • 常用的签名算法包括 RSA、ECDSA（椭圆曲线数字签名算法）等。
  3. 将数字签名附加到证书：
    • 数字签名被附加到证书的签名字段中。
    • 证书的最终形式包括证书内容和数字签名。
  4. 证书的验证：
    • 当客户端或服务器收到证书时，它们可以使用 CA 的公钥验证数字签名。
    • 验证过程包括：
      • 使用 CA 的公钥解密数字签名，得到原始的散列值。
      • 对证书内容重新计算散列值。
      • 比较两个散列值，如果一致，则说明证书未被篡改且是由可信的 CA 签发的。

4.4 使用证书时的通信方式

客户端收到服务器的证书后，进行以下验证：

  • 证书链验证：验证证书是否由受信任的 CA 签发。
  • 证书有效性：验证证书是否在有效期内。
  • 证书域名匹配：验证证书中的域名是否与请求的域名匹配。
  • 证书签名验证：使用 CA 的公钥验证证书的数字签名，确保证书未被篡改。
