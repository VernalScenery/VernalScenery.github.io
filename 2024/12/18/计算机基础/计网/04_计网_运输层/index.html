1 概览

  • 物理层、数据链路层、网络层：解决数据从一个主机传输到另一个主机的问题。
  • 运输层：解决数据从一个主机中的特定进程传输到另一个主机中的特定进程的问题，确保端到端的可靠通信。

进程间的通信

  • 端口号：运输层使用端口号来标识主机上的不同进程。每个进程都有一个唯一的端口号，范围从0到65535。
  • 套接字：运输层通过套接字（Socket）来实现进程间的通信。套接字是由IP地址和端口号组成的唯一标识符，形式为 IP地址:端口号。
  • 连接建立：对于TCP协议，需要通过三次握手建立连接，确保双方都准备好进行通信。
  • 数据传输：运输层协议负责将数据分割成段（Segment），并在接收端重新组装。
  • 连接终止：对于TCP协议，需要通过四次挥手来终止连接，确保数据传输完成且双方都同意断开连接。

3 UDP 协议

UDP（User Datagram Protocol，用户数据报协议）是一种无连接的传输层协议，与TCP相比，UDP提供了更简单的通信机制，但不保证数据的可靠传输。

3.1 主要特点

  1. 无连接：
    • UDP是无连接的协议，发送数据之前不需要建立连接，因此没有连接的开销。
    • 发送方可以直接发送数据报，接收方也可以随时接收数据报。
  2. 尽最大努力交付：
    • UDP不保证数据的可靠传输，不进行重传、排序或错误检测。
    • 如果数据报在传输过程中丢失或损坏，UDP不会进行重传，也不会通知发送方。
  3. 无拥塞控制：
    • UDP没有拥塞控制机制，不会根据网络状况调整数据传输速率。
    • 这可能导致在网络拥塞时，大量数据报被丢弃。
  4. 低延迟：
    • 由于UDP没有连接建立和维护的开销，也没有重传和排序的机制，因此传输延迟较低。
    • 适合对实时性要求较高的应用。
  5. 报文结构简单：
    • UDP报文头只有8个字节，包含源端口号、目的端口号、长度和校验和。
    • 报文头结构简单，处理速度快。

因为传的快，所有如果通过校验和知道传错了，再传一遍就行了

3.2 首部格式

  • 源端口号（Source Port）：2字节，表示发送方的端口号。
  • 目的端口号（Destination Port）：2字节，表示接收方的端口号。
  • 长度（Length）：2字节，表示整个UDP数据报的长度（包括头部和数据部分）。
  • 校验和（Checksum）：2字节，用于检测数据报在传输过程中是否出错。校验和是可选的，如果设置为0，则不进行校验。

4 TCP 协议

TCP（Transmission Control Protocol，传输控制协议）是一种面向连接的、可靠的传输层协议。与UDP不同，TCP提供了数据传输的可靠性、顺序性和流量控制机制，适用于需要高可靠性的网络应用

4.1 主要特点

  1. 面向连接：
    • TCP在数据传输之前需要建立连接，类似于电话通话前的拨号过程。
    • 连接建立后，数据可以在两个方向上可靠地传输。
    • 连接结束后，需要进行连接关闭。
  2. 可靠传输：
    • TCP通过确认应答（ACK）、重传机制、校验和等手段保证数据的可靠传输。
    • 如果数据在传输过程中丢失或损坏，TCP会自动重传数据。
  3. 顺序传输：
    • TCP通过序列号确保数据按顺序到达接收方。
    • 如果数据包乱序到达，TCP会重新排序后再交给应用层。
  4. 流量控制：
    • TCP通过滑动窗口机制进行流量控制，确保发送方不会发送超过接收方处理能力的数据。
    • 这可以防止接收方因处理不过来而导致数据丢失。
  5. 拥塞控制：
    • TCP通过拥塞控制算法（如慢启动、拥塞避免、快重传、快恢复等）动态调整发送速率，避免网络拥塞。
  6. 连接管理：
    • TCP通过三次握手建立连接，通过四次挥手关闭连接。
    • 三次握手确保双方都准备好进行通信。
    • 四次挥手确保数据传输完成后双方都同意断开连接。

4.2 面向字节流

4.2.1 核心思想

  • 应用程序发送的数据被视为一个连续的字节流，而不是固定长度的消息。
  • 发送方和接收方不需要关心数据的具体分界，只需要关注整个字节流的完整性和顺序。

4.2.2 基础概念

  1. 字节流：
    • TCP 将应用程序的数据视为一个无结构的字节流，而不是固定大小的数据报。
    • 发送方可以连续发送多个字节，接收方也会按顺序接收这些字节。
  2. 序列号：
    • 每个字节都会被分配一个唯一的序列号。
    • 序列号从0开始，每发送一个字节，序列号递增1。
    • 例如，如果发送方发送了1000个字节，那么这些字节的序列号分别是0到999。
  3. 数据段：
    • TCP 将字节流分割成多个数据段（Segment）进行传输。
    • 每个数据段包含一个序列号，表示该段数据的第一个字节的序列号。
    • 接收方根据序列号将接收到的数据段重新组合成原始的字节流。

4.2.3 关键机制

  1. 序列号（Sequence Number）：
    • 每个数据段都有一个唯一的序列号，用于标识数据段在字节流中的位置。
    • 发送方为每个字节分配一个序列号，接收方通过确认号告诉发送方已成功接收的字节范围。
  2. 确认号（Acknowledgment Number）：
    • 接收方通过确认号告诉发送方下一个期望接收的字节的序列号。
    • 发送方根据确认号判断哪些数据已经被接收方成功接收，哪些数据需要重传。
  3. 重传机制：
    • 如果发送方在一定时间内没有收到某个数据段的确认，会重新发送该数据段。
    • 通过重传机制，确保数据最终能够可靠地到达接收方。
  4. 流量控制：
    • 接收方通过 TCP 首部中的“窗口大小”字段告知发送方其接收窗口的大小。
    • 发送方根据接收窗口的大小调整发送速率，确保不会发送超过接收方处理能力的数据。
  5. 拥塞控制：
    • 通过拥塞窗口机制，动态调整发送速率，避免网络拥塞。
    • 拥塞窗口的大小根据网络的反馈（如确认消息、丢包等）动态调整。

4.2.4 具体步骤

  1. 数据分割：
    • 发送方将应用程序的数据分割成多个数据段，每个数据段都有一个唯一的序列号。
    • 数据段的大小可以根据网络的最大传输单元（MTU）进行调整。
  2. 数据发送：
    • 发送方根据当前的发送窗口大小发送数据段。
    • 发送方将数据段的序列号记录在发送缓冲区中，等待接收方的确认。
  3. 数据接收：
    • 接收方接收到数据段后，将其存储在接收缓冲区中。
    • 接收方通过确认号和窗口大小字段向发送方发送确认消息。
  4. 确认与重传：
    • 发送方收到确认消息后，根据确认号更新发送缓冲区中的状态。
    • 如果发送方在一定时间内没有收到某个数据段的确认，会重新发送该数据段。
  5. 数据重组：
    • 接收方根据数据段的序列号将数据段重组为原始的字节流。
    • 接收方将重组后的字节流传递给应用程序。

4.3 缓冲区

TCP 协议中的缓冲区主要用于管理数据的发送和接收过程，确保数据传输的效率和可靠性。

4.3.1 核心思想

缓冲区的核心思想在于通过临时存储数据，平滑数据流，确保数据的有序传输和接收方的处理能力。

  1. 平滑数据流：
    • 缓冲区用于临时存储发送方和接收方之间的数据，平滑数据流，避免数据包的丢失和乱序。
    • 发送方可以将数据先写入发送缓冲区，然后按网络条件逐步发送；接收方可以将接收到的数据先存储在接收缓冲区，然后按应用程序的需求逐步读取。
  2. 确保数据的有序传输：
    • 缓冲区帮助维护数据的顺序，确保接收方接收到的数据与发送方发送的数据顺序一致。
    • 通过序列号和确认号机制，TCP 可以检测和纠正数据包的丢失、重复和乱序。
  3. 流量控制：
    • 通过接收缓冲区的大小，接收方可以告知发送方其当前的处理能力，避免发送方发送超过接收方处理能力的数据。
    • 发送方根据接收方的窗口大小调整发送速率，确保接收方不会因缓存溢出而丢弃数据。
  4. 拥塞控制：
    • 通过发送缓冲区的大小，发送方可以动态调整发送速率，避免网络拥塞。
    • 发送方根据网络的反馈（如确认消息、丢包等）调整拥塞窗口的大小，从而控制发送速率。

4.3.2 关键机制

  1. 发送缓冲区（Send Buffer）：
    • 存储发送方尚未发送或尚未被确认的数据。
    • 发送方将应用程序的数据写入发送缓冲区，然后根据网络条件和拥塞窗口的大小逐步发送数据。
    • 发送缓冲区的大小可以根据网络条件和应用程序的需求动态调整。
  2. 接收缓冲区（Receive Buffer）：
    • 存储接收方已经接收到但尚未被应用程序读取的数据。
    • 接收方将接收到的数据存储在接收缓冲区中，然后按应用程序的需求逐步读取数据。
    • 接收缓冲区的大小可以通过 TCP 首部中的“窗口大小”字段告知发送方，从而实现流量控制。
  3. 序列号（Sequence Number）：
    • 每个数据段都有一个唯一的序列号，用于标识数据段在字节流中的位置。
    • 发送方为每个字节分配一个序列号，接收方通过确认号告诉发送方已成功接收的字节范围。
  4. 确认号（Acknowledgment Number）：
    • 接收方通过确认号告诉发送方下一个期望接收的字节的序列号。
    • 发送方根据确认号判断哪些数据已经被接收方成功接收，哪些数据需要重传。
  5. 重传机制：
    • 如果发送方在一定时间内没有收到某个数据段的确认，会重新发送该数据段。
    • 通过重传机制，确保数据最终能够可靠地到达接收方。
  6. 窗口大小（Window Size）：
    • 接收方通过 TCP 首部中的“窗口大小”字段告知发送方其接收窗口的大小。
    • 发送方根据接收窗口的大小调整发送速率，确保不会发送超过接收方处理能力的数据。

4.3.3 具体步骤

  1. 数据写入发送缓冲区：
    • 应用程序将数据写入发送缓冲区。
    • 发送方根据当前的拥塞窗口和接收窗口大小，逐步发送数据。
  2. 数据发送：
    • 发送方从发送缓冲区中取出数据，封装成 TCP 数据段，发送到网络中。
    • 每个数据段都有一个唯一的序列号。
  3. 数据接收：
    • 接收方接收到数据段后，将其存储在接收缓冲区中。
    • 接收方通过确认号告诉发送方已成功接收的字节范围。
  4. 确认与重传：
    • 发送方收到确认消息后，根据确认号更新发送缓冲区中的状态。
    • 如果发送方在一定时间内没有收到某个数据段的确认，会重新发送该数据段。
  5. 数据读取：
    • 应用程序从接收缓冲区中读取数据。
    • 接收方根据应用程序的需求逐步读取数据，确保数据的有序传输。

4.4 首部格式

控制位（Flags）

控制位字段包含6个标志位，每个位表示一个特定的控制功能：

  • **URG (Urgent)**：紧急指针有效，表示数据中有紧急数据。
  • **ACK (Acknowledgment)**：确认号字段有效，表示这是一个确认报文。
  • **PSH (Push)**：提示接收方尽快将数据交付给应用层。
  • **RST (Reset)**：复位连接，用于异常情况下的连接重置。
  • **SYN (Synchronize)**：同步序列号，用于建立连接时的三次握手。
  • **FIN (Finish)**：结束连接，表示发送方已经没有数据发送，请求关闭连接。

选项字段

选项字段是可选的，用于扩展 TCP 的功能。常见的选项包括：

  • **MSS (Maximum Segment Size)**：最大报文段长度，协商双方的最大传输单元。
  • 窗口扩大因子：用于扩展窗口大小字段的范围。
  • 时间戳：用于更精确地控制重传和防止数据包乱序。
  • **NOP (No Operation)**：填充字段，用于对齐选项字段。

4.5 流量控制

4.5.1 核心思想

  • TCP（传输控制协议）的流量控制机制是为了确保发送方不会发送超过接收方处理能力的数据，从而避免接收方因处理不过来而丢弃数据。
  • 通过接收窗口机制，确保发送方不会发送超过接收方处理能力的数据，避免接收方的缓存溢出。

4.5.2 关键概念

  1. 发送窗口（Send Window）：
    • 发送方维护的一个变量，表示可以连续发送但未被确认的数据量。
    • 发送窗口的大小可以根据接收方的反馈动态调整。
  2. 接收窗口（Receive Window）：
    • 接收方维护的一个变量，表示当前接收缓冲区中可以接收的数据量。
    • 接收方通过TCP首部中的“窗口大小”字段告知发送方其接收窗口的大小。
  3. 确认号（Acknowledgment Number）：
    • 接收方通过确认号告诉发送方下一个期望接收的字节的序列号。
    • 发送方根据确认号判断哪些数据已经被接收方成功接收。

4.5.3 流量控制机制

  1. 接收方缓存溢出：
    • 如果接收方的接收缓冲区已满，接收方无法处理更多的数据。
    • 这时，接收方会通过TCP报文段的“窗口”字段告知发送方当前的接收窗口大小。
  2. 接收方调整发送窗口：
    • 接收方会根据自身的缓冲区状态动态调整发送窗口的大小。
    • 如果接收方的缓冲区已满，它会发送一个窗口大小为0的确认消息，要求发送方停止发送数据。
    • 当接收方处理完部分数据并释放缓冲区空间后，会发送一个新的确认消息，告知发送方新的窗口大小。
  3. 发送方调整发送窗口：
    • 发送方根据接收方的窗口大小字段调整自己的发送窗口。
    • 发送窗口的大小由两个因素决定：拥塞窗口（Congestion Window, cwnd）和接收窗口（Receive Window, rwnd）。
    • 发送窗口的计算公式为：发送窗口 = min(拥塞窗口, 接收窗口)。

4.6 拥塞控制

4.6.1 核心思想

  • 通过动态调整拥塞窗口的大小，控制发送窗口的大小，从而避免网络拥塞，确保数据传输的可靠性和高效性
  • 通过拥塞窗口机制，动态调整发送速率，避免网络拥塞，确保网络资源的高效利用。

  1. 避免网络拥塞：
    • 当网络中的链路或路由器过载时，会丢弃数据包，导致网络拥塞。TCP通过拥塞控制机制，动态调整发送方的发送速率，避免网络过载和数据包丢失。
  2. 自适应调整：
    • 发送方根据网络的反馈（如确认消息、丢包等）动态调整拥塞窗口的大小，从而控制发送窗口的大小。发送窗口的大小由拥塞窗口和接收窗口的最小值决定，即 发送窗口 = min(拥塞窗口, 接收窗口)。

4.6.2 关键机制

  1. 拥塞窗口（Congestion Window, cwnd）：
    • 发送方维护的一个变量，表示可以连续发送但未被确认的数据量。
    • 拥塞窗口的大小根据网络的反馈动态调整，以适应网络条件的变化。
  2. 接收窗口（Receive Window, rwnd）：
    • 接收方维护的一个变量，表示当前接收缓冲区中可以接收的数据量。
    • 接收方通过TCP首部中的“窗口大小”字段告知发送方其接收窗口的大小。
  3. 发送窗口：
    • 发送窗口的大小由拥塞窗口和接收窗口的最小值决定，即 发送窗口 = min(拥塞窗口, 接收窗口)。
    • 这样可以确保发送方不会发送超过接收方处理能力的数据，同时也不会超过网络的实际带宽。
  4. 慢启动（Slow Start）：
    • 目的：快速探测网络的带宽。
    • 过程：
      • 初始时，拥塞窗口（cwnd）设置为1个MSS（最大报文段大小）。
      • 每当收到一个确认（ACK），拥塞窗口增加1个MSS。
      • 拥塞窗口呈指数增长，直到达到慢启动阈值（ssthresh）或发生丢包。
  5. 拥塞避免（Congestion Avoidance）：
    • 目的：在接近网络容量时，缓慢增加发送速率，避免拥塞。
    • 过程：
      • 当拥塞窗口达到慢启动阈值（ssthresh）后，进入拥塞避免阶段。
      • 每当收到一个确认（ACK），拥塞窗口增加1/cwnd个MSS。
      • 拥塞窗口呈线性增长。
  6. 快速重传（Fast Retransmit）：
    • 目的：快速检测并重传丢失的数据包。
    • 过程：
      • 如果发送方收到三个重复的确认（ACK），则认为有一个数据包丢失。
      • 立即重传丢失的数据包，而不必等待重传计时器超时。
  7. 快速恢复（Fast Recovery）：
    • 目的：在检测到丢包后，快速恢复拥塞窗口，避免不必要的性能下降。
    • 过程：
      • 当检测到丢包并执行快速重传后，进入快速恢复阶段。
      • 将慢启动阈值（ssthresh）设置为当前拥塞窗口的一半。
      • 将拥塞窗口设置为新的慢启动阈值。
      • 每当收到一个确认（ACK），拥塞窗口增加1个MSS，直到收到一个新的确认（ACK）为止。
      • 然后进入拥塞避免阶段。

4.7 TCP 连接的建立和释放

4.7.1 连接建立

TCP连接的建立采用的是三路握手（Three-Way Handshake）机制，具体步骤如下：

  1. 客户端发送SYN请求：
    • 客户端（主动打开方）向服务器（被动打开方）发送一个SYN（同步）报文段，请求建立连接。
    • SYN报文段中包含一个初始序列号（ISN，Initial Sequence Number）。
    • eg: 客服端SYN = 1, ISN = x
  2. 服务器响应SYN+ACK：
    • 服务器收到SYN报文段后，确认收到请求，并向客户端发送一个SYN+ACK报文段。
    • SYN+ACK报文段中包含服务器的初始序列号（ISN）以及对客户端SYN报文段的确认号（确认号(ack) = 客户端的ISN + 1）。
    • eg: 服务器SYN = 1, 服务器ACK = 1, ISN=y, 确认号(ack) = x + 1
  3. 客户端发送ACK确认：
    • 客户端收到SYN+ACK报文段后，发送一个ACK报文段作为确认。
    • ACK报文段中包含对服务器SYN+ACK报文段的确认号（ACK = 服务器的ISN + 1）。
    • eg: 客服端ACK = 1, ISN= x + 1, 确认号(ack) = y + 1

此时，TCP连接正式建立，客户端和服务器可以开始双向数据传输。

补充一下：

确认号（Acknowledgment Number）：

  • 接收方通过确认号告诉发送方下一个期望接收的字节的序列号。
  • 发送方根据确认号判断哪些数据已经被接收方成功接收。

4.7.2 连接释放

TCP连接的释放采用的是四路挥手（Four-Way Wavehandshake）机制，具体步骤如下：

  1. 客户端发送FIN请求：
    • 客户端向服务器发送FIN报文段，表示不再发送数据，但可以接收数据。
    • 序列号为最后一个发送的数据字节的序列号加1。
  2. 服务器响应ACK确认：
    • 服务器收到FIN报文段后，发送ACK报文段确认收到。
    • 确认号为客户端FIN报文段的序列号加1。
  3. 服务器发送FIN请求：
    • 服务器向客户端发送FIN报文段，表示也不再发送数据，但可以接收数据。
    • 序列号为最后一个发送的数据字节的序列号加1。
  4. 客户端响应ACK确认：
    • 客户端收到服务器的FIN报文段后，发送ACK报文段确认收到。
    • 确认号为服务器FIN报文段的序列号加1。

TIME_WAIT 状态

在客户端发送完最后一个ACK报文段后，客户端会进入 TIME_WAIT 状态。这个状态的持续时间为2倍的MSL（Maximum Segment Lifetime，最大报文段生存时间）。TIME_WAIT 状态的作用包括：

  • 处理延迟的数据包：确保所有可能的延迟数据包都被正确处理。
  • 避免旧数据包干扰新连接：防止旧连接的数据包干扰到新的连接，特别是在两个连接使用相同的源地址、源端口、目的地址和目的端口的情况下。

此时，TCP连接正式关闭，双方都完成了数据的发送和接收。

4.7.3 抓包结果

4.8 延迟ACK

  • 延迟ACK：是指TCP在接收到数据后，并不立即发送ACK报文段，而是等待一段时间或接收到更多的数据后再发送ACK。
  • 目的：减少ACK报文段的数量，降低网络拥塞，提高网络效率。

4.8.1 延迟ACK 的机制

  1. 定时器机制：
    • 当TCP接收到数据后，启动一个定时器（通常是200毫秒）。
    • 如果在这段时间内没有收到更多的数据，定时器到期后会发送ACK报文段。
  2. 数据到达机制：
    • 如果在定时器到期之前，TCP又接收到更多的数据包，它会取消当前的定时器，并重新启动一个新的定时器。
    • 这样可以确保在短时间内接收到多个数据包时，只发送一个ACK报文段。
  3. 最大延迟：
    • 即使没有更多的数据包到达，TCP也会在最大延迟时间（通常是200毫秒）后发送ACK报文段。
    • 这种机制确保了ACK不会无限期延迟，从而避免数据传输的延迟问题。

5 总结

5.1 面向应用报文（Message-Oriented）

5.1.1 定义

  • 面向应用报文 模型将数据视为独立的消息单元（messages），每个消息都有明确的边界。
  • 每个消息在传输过程中被视为一个整体，从发送方到接收方保持完整性和独立性。

5.1.2 特征

  1. 消息完整性：
    • 每个消息在传输过程中保持完整，不会被分割成多个部分。
    • 接收方总是以完整的消息形式接收数据。
  2. 消息边界：
    • 每个消息都有明确的开始和结束边界。
    • 发送方和接收方都需要知道消息的边界，以便正确解析和处理数据。
  3. 可靠性：
    • 通常提供消息级别的可靠性保证，确保每个消息要么完全成功传输，要么完全失败。
    • 可以使用确认机制来确保消息的可靠传输。
  4. 应用场景：
    • 适用于需要明确消息边界的场景，如即时通讯、文件传输、数据库操作等。
    • 常见的协议包括AMQP（Advanced Message Queuing Protocol）、MQTT（Message Queuing Telemetry Transport）等。

5.1.3 示例

假设客户端A向服务器B发送两条消息，每条消息的内容分别为 “Hello” 和 “World”。

  1. 客户端A发送消息：
    • A发送第一条消息 “Hello”。
    • A发送第二条消息 “World”。
  2. 服务器B接收消息：
    • B接收到第一条消息 “Hello”，并将其视为一个独立的消息单元。
    • B接收到第二条消息 “World”，并将其视为另一个独立的消息单元。

5.2 面向字节流（Byte-Stream Oriented）

5.2.1 定义

  • 面向字节流 模型将数据视为连续的字节流（byte stream），没有明确的消息边界。
  • 数据在传输过程中可以被分割成多个部分，接收方需要自行处理这些部分以恢复原始数据。

5.2.2 特征

  1. 无消息边界：
    • 数据在传输过程中没有明确的开始和结束边界。
    • 发送方和接收方不需要知道消息的边界，只需要处理连续的字节流。
  2. 数据分割：
    • 数据可以被分割成多个数据包（packets）进行传输。
    • 接收方需要将这些数据包重新组装成完整的数据流。
  3. 顺序保证：
    • 通常提供字节流级别的顺序保证，确保数据按照发送顺序到达。
    • 接收方需要维护数据的顺序，确保数据流的完整性。
  4. 应用场景：
    • 适用于需要连续数据流的场景，如视频流、音频流、实时通信等。
    • 常见的协议包括TCP（Transmission Control Protocol）等。

5.2.3 示例

假设客户端A向服务器B发送一个包含 “HelloWorld” 的字节流。

  1. 客户端A发送字节流：
    • A发送字节流 “HelloWorld”，可以被分割成多个数据包，例如 “Hel”, “loW”, “orld”。
  2. 服务器B接收字节流：
    • B接收到数据包 “Hel”。
    • B接收到数据包 “loW”。
    • B接收到数据包 “orld”。
    • B将这些数据包重新组装成完整的字节流 “HelloWorld”。
