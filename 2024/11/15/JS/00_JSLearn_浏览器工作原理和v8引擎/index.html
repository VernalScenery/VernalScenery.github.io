浏览器的工作原理

1 整体流程

1.1 用户输入 URL

用户在浏览器地址栏输入 URL（例如 https://www.example.com），按下回车键后，浏览器开始处理请求。

1.2 DNS 解析

  • DNS（Domain Name System）：将人类可读的域名（如 www.example.com）转换为计算机可读的 IP 地址（如 192.0.2.1）。
  • 步骤：
    1. 浏览器检查本地缓存是否有该域名的 IP 地址。
    2. 如果没有，浏览器会查询操作系统的 DNS 缓存。
    3. 如果操作系统也没有，浏览器会发送 DNS 查询请求到指定的 DNS 服务器。
    4. DNS 服务器返回 IP 地址。

1.3 建立 TCP 连接

  • TCP（Transmission Control Protocol）：一种可靠的传输协议，用于在客户端和服务器之间建立连接。
  • 步骤：
    1. 浏览器向服务器的 IP 地址和端口（默认是 80 对于 HTTP，443 对于 HTTPS）发送一个 SYN（同步）包。
    2. 服务器响应一个 SYN-ACK（同步确认）包。
    3. 浏览器发送一个 ACK（确认）包，完成三次握手，建立 TCP 连接。

1.4 发送 HTTP 请求

  • HTTP（Hypertext Transfer Protocol）：用于从 Web 服务器请求网页的协议。
  • 步骤：
    1. 浏览器通过已建立的 TCP 连接发送一个 HTTP GET 请求到服务器。
    2. 请求中包含请求行（例如 GET / HTTP/1.1）、请求头（例如 Host: www.example.com）和请求体（如果有的话）。

1.5 服务器处理请求

  • Web 服务器：接收 HTTP 请求，处理请求并生成响应。
  • 步骤：
    1. 服务器解析请求，确定请求的资源（例如 HTML 文件、图片等）。
    2. 服务器处理请求，可能涉及数据库查询、文件读取等操作。
    3. 服务器生成 HTTP 响应，包含响应状态行（例如 HTTP/1.1 200 OK）、响应头（例如 Content-Type: text/html）和响应体（例如 HTML 内容）。

1.6 接收 HTTP 响应

  • 步骤：
    1. 浏览器接收服务器返回的 HTTP 响应。
    2. 浏览器解析响应头，确定响应的内容类型和长度。
    3. 浏览器解析响应体，通常是 HTML 文档。

1.7 渲染页面

  • 渲染引擎：负责解析 HTML、CSS 和 JavaScript，构建 DOM 树和 CSSOM 树，生成渲染树，布局和绘制页面。
  • 步骤：
    1. 解析 HTML：构建 DOM 树。
    2. 解析 CSS：构建 CSSOM 树。
    3. 生成渲染树：结合 DOM 树和 CSSOM 树，生成渲染树。
    4. 布局：计算每个节点的位置和大小。
    5. 绘制：将渲染树绘制到屏幕上，显示页面内容。

1.8 加载资源

  • 资源加载：浏览器会继续加载页面中引用的其他资源，如图片、脚本、样式表等。
  • 步骤：
    1. 解析 HTML 文档，发现 <img>、<script>、<link> 等标签。
    2. 发送 HTTP 请求获取这些资源。
    3. 接收并处理这些资源，更新页面内容。

1.9 执行 JavaScript

  • JavaScript 引擎：解析和执行页面中的 JavaScript 代码。
  • 步骤：
    1. 解析 JavaScript 代码，生成抽象语法树（AST）。
    2. 将 AST 编译成字节码或机器码。
    3. 执行 JavaScript 代码，可能涉及 DOM 操作、网络请求等。

1.10 页面交互

  • 事件循环：JavaScript 是单线程的，使用事件循环处理异步操作和用户交互。
  • 步骤：
    1. 监听用户事件（如点击、滚动等）。
    2. 处理事件队列中的任务。
    3. 更新页面内容，响应用户操作。

1.11 总结

  1. 用户输入 URL：用户在浏览器地址栏输入 URL。
  2. DNS 解析：将域名转换为 IP 地址。
  3. 建立 TCP 连接：通过三次握手建立 TCP 连接。
  4. 发送 HTTP 请求：浏览器发送 HTTP GET 请求。
  5. 服务器处理请求：服务器处理请求并生成响应。
  6. 接收 HTTP 响应：浏览器接收服务器返回的响应。
  7. 渲染页面：浏览器解析 HTML、CSS 和 JavaScript，构建和绘制页面。
  8. 加载资源：浏览器继续加载页面中引用的其他资源。
  9. 执行 JavaScript：浏览器解析和执行 JavaScript 代码。
  10. 页面交互：浏览器处理用户事件和异步操作。

浏览器数据请求的逻辑

先请求服务器下载到 index.html, 然后再对 index.html所引用的 css 和 js文件进行下载

然后浏览器内核对数据进行解析

2 浏览器内核

事实上，我们经常说的浏览器内核指的是浏览器的排版引擎：

排版引擎（layout engine），也称为浏览器引擎（browser engine）、页面渲染引擎（rendering engine）或样版引擎。

3 浏览器渲染过程

Html 和 Css的解析过程

DOM Tree 结合 Style Rules生成 Render Tree

通过 Layout（布局）引擎对 Render Tree 进行一些具体的操作，因为每个浏览器页面的具体情况不一样

最后在通过 Render Tree 渲染到网页上

这一步可能会有 JS 代码对 DOM 进行操作

会停止解析 HTML，而去加载和执行 JavaScript 代码；那么，JavaScript 代码由谁来执行呢？

4 JS 引擎

高级的编程语言都是需要转成最终的机器指令来执行的
事实上我们编写的 JavaScript 无论你交给浏览器或者 Node执行，最后都是需要被CPU 执行的；
但是 CPU 只认识自己的指令集，实际上是机器语言，才能被 CPU 所执行；
所以我们需要JavaScript 引擎帮助我们将JavaScript 代码翻译成CPU 指令来执行；

5 浏览器内核和 JS 引擎的关系

以 WebKit 为例，WebKit 事实上由两部分组成的：

  • WebCore：负责 HTML 解析、布局、渲染等等相关的工作；
  • JavaScriptCore：解析、执行 JavaScript 代码；

即浏览器内核包括 JS 引擎

V8引擎解析JS代码的流程

V8 引擎是 Google 开发的高性能 JavaScript 引擎，广泛用于 Chrome 浏览器和 Node.js 等平台。V8 引擎的解析和执行 JavaScript 代码的流程可以分为几个主要阶段：词法分析、语法分析、编译、优化和执行

官方对V8引擎的定义

  • V8是用C ++编写的Google开源高性能JavaScript和WebAssembly引擎，它用于Chrome和Node.js等。
  • 它实现ECMAScript和WebAssembly(浏览器的下一代语言)
  • V8可以独立运行，也可以嵌入到任何C ++应用程序中

1.4 优化（Optimization）

  • 即时编译器（Just-In-Time Compiler，JIT）：在运行时对字节码进行优化，生成高效的机器码。
  • 内联缓存（Inline Caching）：缓存方法调用的结果，减少重复计算。
  • 类型推断（Type Inference）：推断变量的类型，减少类型检查的开销。

1.5 执行（Execution）

  • 解释器（Interpreter）：逐行执行字节码。
  • 执行上下文（Execution Context）：管理变量、函数和作用域。
  • 调用栈（Call Stack）：记录函数调用的顺序。

1.5.1 示例

执行过程：

  1. 创建全局执行上下文。
  2. 解析并执行全局代码，调用 add 函数。
  3. 创建 add 函数的执行上下文，执行函数体。
  4. 返回结果，更新全局执行上下文中的 result 变量。
  5. 调用 console.log，输出结果。

1.6 总结

  1. 词法分析：将源代码分解成标记。
  2. 语法分析：将标记序列转换成抽象语法树（AST）。
  3. 编译：将 AST 转换为字节码或机器码。
  4. 优化：使用 JIT 编译器对字节码进行优化，生成高效的机器码。
  5. 执行：解释器逐行执行字节码，管理执行上下文和调用栈。

2 其他

1、Parse 主要 包括对 JS 源代码 进行 词法分析 和 语法分析

2、ignition 再对 AST 抽象语法树 进行解析生成字节码，再将字节码针对不同平台转换为机器码(字节码方便跨平台)

不直接生成机器码是因为 cpu 的架构不同，能执行的机器指令是不一样的

3、对于一些需要重复调用的代码，会进行标记hot，然后将这些标记的代码直接转换为机器码，这样下次调用直接执行机器码就行了，会 大大提高效率

4、如下函数被多次调用，被标记为hot , 第一次执行时会解析为机器码，这样第二次调用时直接执行机器码即可，但是由于 JS 的动态类型，第三次传的参数为字符串，之前第一次解析的机器码没法使用，这时，v8引擎会对第一次解析的机器码进行Deoptimization(动态反优化)生成字节码，再执行。这样会有点浪费性能，故 TS 编译的 JS 代码执行效率会高一些
