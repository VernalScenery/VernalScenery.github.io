1 浏览器数据请求的逻辑

先请求服务器下载到 index.html, 然后再对 index.html所引用的 css 和 js文件进行下载

然后浏览器内核对数据进行解析

2 浏览器内核

事实上，我们经常说的浏览器内核指的是浏览器的排版引擎：

排版引擎（layout engine），也称为浏览器引擎（browser engine）、页面渲染引擎（rendering engine）或样版引擎。

3 浏览器渲染过程

Html 和 Css的解析过程

DOM Tree 结合 Style Rules生成 Render Tree

通过 Layout（布局）引擎对 Render Tree 进行一些具体的操作，因为每个浏览器页面的具体情况不一样

最后在通过 Render Tree 渲染到网页上

这一步可能会有 JS 代码对 DOM 进行操作

会停止解析 HTML，而去加载和执行 JavaScript 代码；那么，JavaScript 代码由谁来执行呢？

4 JS 引擎

高级的编程语言都是需要转成最终的机器指令来执行的
事实上我们编写的 JavaScript 无论你交给浏览器或者 Node执行，最后都是需要被CPU 执行的；
但是 CPU 只认识自己的指令集，实际上是机器语言，才能被 CPU 所执行；
所以我们需要JavaScript 引擎帮助我们将JavaScript 代码翻译成CPU 指令来执行；

5 浏览器内核和 JS 引擎的关系

以 WebKit 为例，WebKit 事实上由两部分组成的：

  • WebCore：负责 HTML 解析、布局、渲染等等相关的工作；
  • JavaScriptCore：解析、执行 JavaScript 代码；

即浏览器内核包括 JS 引擎

6 V8 引擎的原理

6.1 官方对 V8 引擎的定义

  • V8 是用C ++编写的 Google 开源高性能 JavaScript 和 WebAssembly 引擎，它用于 Chrome 和 Node.js 等。
  • 它实现ECMAScript和WebAssembly(浏览器的下一代语言)
  • V8 可以独立运行，也可以嵌入到任何 C ++应用程序中

6.2 V8 引擎解析 JS 代码的流程

1、Parse 主要 包括对 JS 源代码 进行 词法分析 和 语法分析

eg:

先进行词法分析

再通过语法分析 生成 AST 抽象语法树，相关网站 ast explorer

解析 ts 和 vue 模板都用到了 AST

2、ignition 再对 AST 抽象语法树 进行解析生成字节码，再将字节码针对不同平台转换为机器码(字节码方便跨平台)

不直接生成机器码是因为 cpu 的架构不同，能执行的机器指令是不一样的

3、对于一些需要重复调用的代码，会进行标记hot，然后将这些标记的代码直接转换为机器码，这样下次调用直接执行机器码就行了，会 大大提高效率

4、如下函数被多次调用，被标记为hot , 第一次执行时会解析为机器码，这样第二次调用时直接执行机器码即可，但是由于 JS 的动态类型，第三次传的参数为字符串，之前第一次解析的机器码没法使用，这时，v8 引擎会对第一次解析的机器码进行Deoptimization(动态反优化)生成字节码，再执行。这样会有点浪费性能，故 TS 编译的 JS 代码执行效率会高一些

7 JavaScript 的执行过程

7.1 编译阶段

JS 引擎会在执行代码之前，会在堆内存中创建一个全局对象：Global Object（GO）

  • 该对象所有的作用域（scope） 都可以访问；
  • 里面会包含Date、Array、String、Number、setTimeout、setInterval等等；
  • 其中还有一个window 属性指向自己；

这个 this指向的就是 GlobalObject
全局定义的变量会加入到GlobalObject中，但是并不会赋值
全局定义的函数会进行预处理，函数名会保存对应函数存储空间的地址，而在函数存储空间中会保存函数的父级作用域和函数执行体(代码块)

注意：编译阶段函数的父级作用域就确定了，而非执行函数时确定

7.2 执行上下文栈（调用栈）

JS 引擎内部有一个执行上下文栈（Execution Context Stack，简称 ECS），它是用于执行代码的调用栈。
某一上下文(代码块) 执行时入栈，执行完出栈

7.3 执行阶段——全局执行上下文(GEC)

首先执行的是全局的代码块

  • 全局的代码块为了执行会构建一个 全局执行上下文 Global Execution Context（GEC） ；
  • GEC 会被放入到 ECS 中 执行

GEC被放入到 ECS 中里面包含两部分内容：

  • 第一部分：在代码执行前(编译阶段)，parser 转成 AST的过程中，会将全局定义的变量、函数等加入到GlobalObject中，但是并不会赋值；这个过程也称之为变量的作用域提升（hoisting）
  • 第二部分：在**代码执行中(执行阶段)**，对变量赋值，或者执行其他的函数；

如下代码 console.log(num1)打印的就是undefind

7.4 执行阶段——函数执行上下文(FEC)

执行到一个函数时，就会根据函数体创建一个函数执行上下文（Functional Execution Context，简称 FEC），并且压入到EC Stack中。

FEC中包含三部分内容：

  • 第一部分：在代码执行前(编译阶段)，parser 转成 AST的过程中，会创建一个Activation Object（AO）,
    AO中包含形参、arguments、函数定义和指向函数对象的指针、定义的变量；
  • 第二部分：作用域链：由 VO（在函数中就是 AO 对象）和父级 VO 组成，查找时会一层层查找；
  • 第三部分：this 绑定的值
